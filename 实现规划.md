### 1.2 当前架构判定：**这是一个纯前端 Chatbot，不是 Agent**

| 维度 | Chatbot（当前） | Agent（目标） |
|------|:--------------:|:------------:|
| **调用模式** | 单轮请求-响应 | 多步规划-执行-观察-反思循环 |
| **工具/函数调用** | ❌ 无 | ✅ 结构化工具调用 |
| **自主决策** | ❌ 硬编码规则（关键词匹配判断追问） | ✅ LLM 自主决定下一步动作 |
| **记忆系统** | ❌ 仅最近 10 轮对话历史 | ✅ 工作记忆 + 情景记忆 + 候选人画像 |
| **输出结构** | 自由文本（带图片时 JSON） | 结构化决策（动作 + 参数 + 推理） |
| **能力边界** | 仅限单次 LLM 回复 | 可调用外部工具扩展能力 |
| **错误恢复** | 固定降级文案 | 根据错误类型自主选择替代方案 |

**核心问题**：当前 `InterviewAI.processTurn()` 的本质是：

```
用户输入 → 拼装 messages 数组 → 调用 LLM API → 返回文本
```

这是一次性的、无状态的函数调用。AI 面试官无法：
- 主动决定"这个回答不够深入，我需要从另一个角度追问"
- 调用评分工具对当前回答打分，根据分数决定下一步
- 维护候选人的动态画像（优势领域、薄弱环节、情绪变化曲线）
- 在面试结束时自主汇总生成结构化报告

## 二、Agent 化改造目标

### 2.1 核心愿景

将当前的"带 System Prompt 的问答机器人"改造为**具备自主规划、工具使用、多步推理能力的 AI Agent 面试官**。

### 2.2 Agent 面试官应具备的能力

1. **自主规划面试策略**：根据简历+JD，动态生成面试计划，而非固定题库顺序
2. **工具调用能力**：调用评分工具、知识检索工具、报告生成工具等
3. **多步推理**：观察回答质量 → 分析薄弱点 → 决定追问方向 → 执行追问
4. **长期记忆**：维护候选人画像，跨问题追踪候选人的表现趋势
5. **自主收尾**：判断信息收集充分后，自主决定结束面试并生成报告

### 2.3 与现有系统的关系

Agent 层是**对现有服务层的封装和编排**，而非重写：

```
┌─────────────────────────────────────────────────┐
│                  Agent 层（新增）                  │
│  ┌──────────┐ ┌──────────┐ ┌────────────────┐   │
│  │ Planner  │ │ Executor │ │ Memory Manager │   │
│  └────┬─────┘ └────┬─────┘ └───────┬────────┘   │
│       │            │               │             │
│  ┌────┴────────────┴───────────────┴─────────┐   │
│  │            Tool Registry                  │   │
│  │  ask_question │ evaluate_answer │ ...    │   │
│  └──────────────────┬───────────────────────┘   │
└─────────────────────┼───────────────────────────┘
                      │
┌─────────────────────┼───────────────────────────┐
│              现有服务层（复用）                    │
│  llm.ts │ speech.ts │ conversation.ts │ stt.ts   │
│  interview-engine.ts │ scoring.ts │ prompts.ts   │
└─────────────────────────────────────────────────┘
```

### 3.2 Agent 主循环设计

```
┌──────────────────────────────────────────────────┐
│                 Agent 主循环                       │
│                                                  │
│   ┌─────────┐                                    │
│   │  PLAN   │ ← 分析当前状态，生成下一步计划        │
│   └────┬────┘                                    │
│        │ action + params                         │
│        ▼                                         │
│   ┌─────────┐                                    │
│   │  ACT    │ ← 调用对应工具，执行动作              │
│   └────┬────┘                                    │
│        │ observation (工具返回结果)                │
│        ▼                                         │
│   ┌─────────┐                                    │
│   │ OBSERVE │ ← 处理工具结果，更新记忆             │
│   └────┬────┘                                    │
│        │ reflection (对当前状态的判断)              │
│        ▼                                         │
│   ┌─────────┐                                    │
│   │ REFLECT │ ← 评估进展，决定继续还是结束          │
│   └────┬────┘                                    │
│        │                                          │
│   ┌────┴────┐                                     │
│   │ 继续？   │── yes ──→ PLAN                     │
│   └─────────┘                                     │
│        │ no                                       │
│        ▼                                          │
│   生成最终报告                                     │
└──────────────────────────────────────────────────┘
```

### 3.3 工具（Tools）设计

每个工具遵循统一接口：

```typescript
interface AgentTool {
  name: string;                    // 工具名称（传给 LLM）
  description: string;             // 工具描述（LLM 用于判断何时调用）
  parameters: ToolParameter[];     // 参数定义
  execute(params: Record<string, unknown>): Promise<ToolResult>;
}

interface ToolResult {
  success: boolean;
  data?: unknown;                  // 成功时的返回数据
  error?: string;                  // 失败时的错误信息
  cost?: { tokens: number };       // 成本追踪
}
```

#### 工具清单

| 工具名 | 用途 | 调用 LLM？ | 副作用 |
|--------|------|:---------:|:------:|
| `ask_question` | 向候选人提出问题（TTS 播报） | 否 | 推送 UI 消息 |
| `evaluate_answer` | 评估当前回答质量（4 维度评分） | 是 | 更新候选人画像 |
| `generate_followup` | 针对回答薄弱点生成追问 | 是 | 无 |
| `search_knowledge` | 检索技术知识（联网或本地题库） | 可选 | 无 |
| `analyze_vision` | 分析摄像头画面（作弊/情绪） | 是 | 更新监控状态 |
| `update_profile` | 更新候选人能力/情绪画像 | 是 | 写入记忆 |
| `generate_report` | 汇总生成面试报告 | 是 | 写入历史记录 |
| `control_flow` | 决定：下一题/追问/跳过/结束 | 是 | 推进面试流程 |

### 3.4 记忆系统设计

#### 工作记忆（Working Memory）
- 当前面试的即时上下文
- 内容：最近 5 轮对话 + 当前问题 + 当前候选回答
- 生命周期：当前面试会话
- 存储：内存（不持久化）

#### 候选人画像（Candidate Profile）
- 动态更新的候选人能力模型
- 内容：每个维度的得分趋势、强项/弱项标签、情绪状态历史
- 示例：
```json
{
  "strengths": ["算法基础", "项目经验"],
  "weaknesses": ["系统设计", "沟通表达"],
  "dimensionScores": {
    "communication": 7.5,
    "professionalKnowledge": 8.0,
    "logicalThinking": 6.5,
    "adaptability": 7.0
  },
  "emotionTrend": ["nervous", "nervous", "calm", "confident"],
  "cheatingFlags": 0
}
```
- 生命周期：当前面试会话（结束后归档至历史记录）
- 存储：内存 → 结束时写入 localStorage

#### 情景记忆（Episodic Memory）
- 面试中的关键事件快照
- 内容：显著的正面/负面表现、违规事件、转折点
- 用于最终报告生成和长期趋势分析

### 3.5 与 LLM 的交互协议

Agent 循环中的每次 LLM 调用使用 **工具调用（Tool Use / Function Calling）** 格式：

```
System: 你是一个 AI 面试官 Agent。你可以调用以下工具来执行面试：
  - ask_question(question, type, difficulty)
  - evaluate_answer(answer, question, criteria)
  - generate_followup(gap_area, original_question)
  - control_flow(action: "next_question" | "follow_up" | "skip" | "end")
  ...

User: [候选人回答内容]

Assistant (Tool Call): evaluate_answer(
  answer="我理解 React 的虚拟 DOM...",
  question="请解释 React 虚拟 DOM 的工作原理",
  criteria={...}
)

Tool Result: { score: 7.5, gaps: ["diff算法细节不足"], completeness: 0.6 }

Assistant (Tool Call): generate_followup(
  gap_area="diff算法细节",
  original_question="请解释 React 虚拟 DOM 的工作原理"
)

Tool Result: { followup: "你提到了虚拟 DOM，能详细说说 React 的 diff 算法具体是如何工作的吗？..." }

Assistant: [使用 ask_question 工具发出追问]
```

### 3.6 技术选型决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| Agent 框架 | **自研轻量 Agent 循环** | 项目已支持 6 家 LLM 服务商，引入 LangChain 等框架会增加依赖体积且与部分服务商（中国厂商）兼容性差 |
| LLM 交互协议 | **OpenAI Function Calling 兼容格式** | 所选服务商均支持此格式；Anthropic 通过 `x-api-key` 头已有适配 |
| 记忆存储 | **内存 + localStorage** | 保持纯前端架构，面试数据不离开浏览器 |
| 工具实现 | **TypeScript 类 + 注册表模式** | 类型安全、可测试、易扩展 |
| 流式响应 | **支持流式 + 非流式双模式** | 流式提升交互体验（边说边显示），非流式用于批量工具调用 |

## 五、技术风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| **浏览器端 Agent 循环延迟** | 每次工具调用需等待 LLM 响应，多步循环可能让用户等待过久 | 流式响应 + 乐观 UI 更新 + 步数上限控制 |
| **LLM 工具调用格式不一致** | 不同服务商的 function calling 实现差异 | 统一适配层 + 各服务商格式测试 |
| **Agent 循环失控** | LLM 可能陷入无效循环（反复追问同一问题） | 最大步数硬限制 + 循环检测 + 降级回经典模式 |
| **API Key 浏览器端安全** | 纯前端架构，API Key 在浏览器中 | 保持 localStorage 存储 + 未来可选后端代理 |
| **Agent 模式成本高于经典模式** | 多步 LLM 调用增加 token 消耗 | 成本追踪透明化 + 工具调用合并 + 可切换经典模式 |

// src/types/agent.ts

/** Agent 当前状态 */
type AgentState = 'idle' | 'planning' | 'acting' | 'waiting_for_user' | 'observing' | 'reflecting' | 'done';

/** Agent 决策结果 */
interface AgentDecision {
  action: string;               // 要调用的工具名
  params: Record<string, unknown>;
  reasoning: string;            // LLM 的推理过程
  confidence: number;           // 决策置信度 0-1
}

/** Agent 观察结果 */
interface AgentObservation {
  toolName: string;
  toolResult: ToolResult;
  memoryUpdates: MemoryUpdate[];
}

/** 记忆更新 */
interface MemoryUpdate {
  type: 'working' | 'candidate_profile' | 'episodic';
  operation: 'add' | 'update' | 'delete';
  data: Record<string, unknown>;
}

/** Agent 工具定义 */
interface AgentTool {
  name: string;
  description: string;
  parameters: {
    name: string;
    type: 'string' | 'number' | 'boolean' | 'object' | 'array';
    description: string;
    required: boolean;
    enum?: string[];
  }[];
  execute(params: Record<string, unknown>, context: AgentContext): Promise<ToolResult>;
}

/** Agent 上下文（注入到每个工具） */
interface AgentContext {
  workingMemory: WorkingMemory;
  candidateProfile: CandidateProfile;
  llmClient: LLMClient;
  config: AgentConfig;
}
