# 贡献规范

## 项目评审约束（必须遵守）

- 每个 PR 只包含一个独立功能，不要混入其他修改
- 提供完整的 PR 描述（标题、功能描述、实现思路、测试方式）
- commit 信息必须用 feat:/fix:/docs:/chore: 开头
- 所有代码应能独立运行，不依赖本地未提交的文件

## 全局规则

本次项目所有开发者可见内容（包括 commit message、PR 描述、README、代码注释、文档）都必须使用中文。

## Commit 规范

遵循 Conventional Commits：

```
feat: 添加摄像头采集 Hook
fix: 修复组件卸载时资源泄漏
docs: 更新 README 安装步骤
chore: 配置 ESLint 规则
refactor: 提取公共工具函数
```

## PR 提交规范

### 单一功能原则

- 每个 PR 只实现或修改单一功能
- 大功能拆分为多个独立 PR 分步提交
- PR 代码变更量适度（通常不超过 300 行新增）

### PR 标题与描述（四项缺一不可）

1. **标题**：一句话说明本 PR 新增/修改了什么，使用中文
2. **功能描述**：说明该功能的作用与使用方式
3. **实现思路**：简要说明技术选型或核心实现逻辑
4. **测试方式**：如何验证该功能正常运行

### 主分支可运行性

- PR 合并后，主分支代码需保持可运行状态
- 评委在任意时间查看应能复现演示效果

## 分支策略

- `master`：保护分支，始终可运行
- feature 分支：每个 PR 独立分支，合并后删除
- 分支命名：`feature/N` 或 `pr/N-功能描述`

## 代码风格

- TypeScript 严格模式
- ESLint Flat Config + Prettier 自动检查
- 运行 `npm run lint` 和 `npm run format:check` 确保通过
- 注释使用中文，关键逻辑和复杂算法有清晰注释

## 开发流程

1. 从 `master` 创建 feature 分支
2. 在 feature 分支上开发并提交
3. 推送到 GitHub 并创建 Pull Request
4. PR 通过 review 后合并到 master
5. 删除 feature 分支

## 无效情形（绝对禁止）

- PR 描述空白或与实际代码变更严重不符
- 引用了第三方库或框架，没在 README 中列明依赖，且未说明原创功能部分
- 复用了自己过去的代码片段，没在 PR 描述中注明来源
